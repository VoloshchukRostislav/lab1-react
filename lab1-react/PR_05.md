# Реалізація компонентів

## 1. Компонент ActivityItem

Файл: `src/components/ActivityItem.jsx`

```javascript
import React from "react";

const ActivityItem = ({ action, date }) => {
  return (
    <div style={{ borderBottom: "1px solid #eee", padding: "10px 0" }}>
      <p style={{ margin: 0, fontWeight: "bold" }}>{action}</p>
      <small style={{ color: "gray" }}>{date}</small>
    </div>
  );
};

export default ActivityItem;
```

Цей компонент є **презентаційним** і відповідає за відображення однієї активності користувача.

# 3. Компонент ActivityList

Файл: `src/components/ActivityList.jsx`

```javascript
import React from "react";
import ActivityItem from "./ActivityItem";

const ActivityList = ({ activities }) => {
  if (!activities || activities.length === 0) {
    return <p>Немає останніх активностей.</p>;
  }

  return (
    <div>
      <h3>Остання активність</h3>

      {activities.map((item) => (
        <ActivityItem
          key={item.id}
          action={item.action}
          date={item.date}
        />
      ))}

    </div>
  );
};

export default ActivityList;
```

Компонент відповідає за відображення списку активностей користувача.

# 4. Контейнерний компонент ProfilePage

Файл: `src/pages/ProfilePage.jsx`

```javascript
import React, { useState, useEffect } from "react";

import UserInfo from "../components/UserInfo";
import ActivityList from "../components/ActivityList";

const ProfilePage = () => {

  const [userData, setUserData] = useState(null);
  const [activities, setActivities] = useState([]);

  useEffect(() => {

    setUserData({
      name: "Іван Бойко",
      email: "ivan.boyko@example.com",
      role: "Frontend Розробник",
    });

    setActivities([
      { id: 1, action: "Увійшов у систему", date: "Сьогодні, 10:00" },
      { id: 2, action: "Оновив профіль", date: "Вчора, 14:30" },
      { id: 3, action: "Завантажив звіт", date: "12 Вересня" },
    ]);

  }, []);

  if (!userData) {
    return <div>Завантаження профілю...</div>;
  }

  return (
    <div style={{ display: "flex", gap: "20px", padding: "20px" }}>

      <aside style={{ flex: 1 }}>
        <UserInfo
          name={userData.name}
          email={userData.email}
          role={userData.role}
        />
      </aside>

      <main style={{ flex: 2 }}>
        <ActivityList activities={activities} />
      </main>

    </div>
  );
};

export default ProfilePage;
```

Цей компонент є **контейнерним (Smart)**, оскільки він керує станом і передає дані дочірнім компонентам через props.

# Контрольні запитання

## 1. Які основні відмінності між презентаційним (Dumb) та контейнерним (Smart) компонентами?

Презентаційні компоненти відповідають лише за відображення інтерфейсу. Вони отримують дані через props і не містять складної логіки.

Контейнерні компоненти керують станом додатку, виконують бізнес-логіку та передають дані презентаційним компонентам.

---

## 2. Поясніть принцип Single Source of Truth

Принцип "Single Source of Truth" означає, що кожна частина стану повинна зберігатися лише в одному місці.

Це дозволяє уникнути дублювання даних, спрощує синхронізацію стану та полегшує керування додатком.

Зазвичай стан зберігається у батьківському компоненті, який передає дані дочірнім компонентам через props.

---

## 3. Чому функція map() зазвичай знаходиться у композиційних компонентах?

Функція `.map()` використовується для ітерації по масиву та створення списку компонентів.

Тому вона зазвичай знаходиться у композиційних компонентах (наприклад ActivityList), які працюють із масивами даних.

Атомарні компоненти (наприклад ActivityItem) відповідають лише за відображення одного елемента.

---

## 4. Що таке Prop Drilling?

Prop Drilling — це передача даних через багато рівнів компонентів за допомогою props.

Це може ускладнювати структуру додатку, коли дані передаються через компоненти, які самі не використовують ці дані.

У таких випадках часто використовують Context API або бібліотеки керування станом.

---

## 5. Критерії створення окремого компонента

Окремий компонент створюють у таких випадках:

* коли частина JSX повторюється
* коли компонент має власну логіку
* коли потрібно перевикористання компонента
* коли великий компонент потрібно розбити для кращої читабельності
* для дотримання принципу розподілу відповідальності
